2. 需求背景与痛点

    分类缺失： 长期以来，Jira 任务单缺乏规范的标签或模块分类，导致无法复盘高频问题。

    语义模糊： 传统的关键词搜索无法识别“数据库连接失败”与“DB Connection Timeout”其实是同一类问题。

    人工成本： 手工为成千上万条历史数据补打标签不切现实。

2.1 行业现状分析

    市场调研显示：
    • 78%的企业面临项目管理数据孤岛问题
    • 平均每个企业Jira项目库积累5-50万条历史任务
    • 传统分类方法准确率仅为45-60%
    • 数据治理投入占IT预算的15-25%
    
2.2 竞品分析

    主要竞品对比：
    • Jira原生标签系统：功能基础，缺乏智能推荐
    • 第三方分类插件：准确率有限，定制化不足
    • 传统BI工具：无法处理非结构化文本语义
    • 通用AI服务：缺乏领域专业知识和上下文理解

3. 核心功能模块 (Phase 1：历史数据治理)
3.1 模块 A：智能数据预处理 (Data Preprocessing)

    最佳实践要求：
    • 数据质量监控：实时监控数据清洗效果，建立质量评分体系
    • 可追溯性：保留原始数据和清洗过程的完整审计日志
    • 可配置性：支持用户自定义清洗规则和参数
    • 性能优化：采用流式处理和并行计算提升处理速度

    输入： 支持 CSV/JSON 格式的 Jira 导出数据（含 Summary, Description），支持自定义。

    处理逻辑：

        噪声剥离： 利用正则或启发式算法过滤掉 Description 中的大段 Log 日志、Java/Python 堆栈信息、Base64 图片编码，防止其干扰语义中心。

        语义合成： 提取 Summary 的全部文本与 Description 清洗后的前 500 个字符，合成“特征文本”。

    输出： 结构化的干净数据集。

3.2 模块 B：向量存储管理 (Vector Store via LanceDB)

    企业级特性：
    • 数据分区：按时间、项目维度进行数据分区管理
    • 压缩优化：采用量化压缩技术减少存储空间50%以上
    • 备份恢复：支持增量备份和快速恢复机制
    • 监控告警：实时监控存储健康状态和性能指标

    核心引擎： 采用 LanceDB 作为底层存储。

    向量化： 调用 Embedding 模型（如 text-embedding-3-small 或国产 BGE-M3）将特征文本转化为高维向量。

    索引构建： 在 LanceDB 中构建向量表，存储 issue_id, vector, raw_text, created_at 等元数据。

        技术特性： 利用 LanceDB 的磁盘列式存储特性，确保即便数据量达到 10w+ 也能秒级检索。

3.3 模块 C：无监督语义聚类 (Unsupervised Clustering)

    算法优化策略：
    • 参数自适应：基于数据特征自动调整聚类参数
    • 多尺度分析：支持不同粒度的聚类层次
    • 增量学习：支持新数据的在线学习和模型更新
    • 可解释性：提供聚类结果的可解释性分析

    算法选择： 优先使用 HDBSCAN 或 K-Means。

    聚类逻辑：

        从 LanceDB 加载向量。

        根据语义密度自动划分类别（簇）。

        簇标签生成（LLM-Assisted）： 对每个簇，随机抽取距离中心点最近的 5 条样本，提交给 LLM 总结出该类的“业务主题”（例如：系统登录异常、UI 布局优化、性能压测反馈）。

    输出： 聚类结果报告，包含每个 Issue ID 所属的新分类。

4. 业务用例故事 (User Stories)
4.1 场景一：存量任务的“语义大扫除”

    用户动作： 管理员导入过去两年的 5000 条 Jira 导出文件。

    系统逻辑： 系统自动完成向量化并执行聚类。

    预期结果： 系统反馈：“发现 12 个核心问题类型，其中 30% 为‘数据库超时’相关，20% 为‘前端样式兼容性’相关。”
    
4.2 场景二：人工复核确认流程
    
    企业级复核机制：
    • 权限分级：不同角色具有不同的复核权限和责任
    • 审批流程：支持多级审批和会签机制
    • 版本控制：复核决策的历史版本管理和回滚能力
    • 合规要求：满足企业内部审计和合规性要求
    
    用户动作： 系统完成自动聚类后，进入人工复核界面。
    
    核心复核功能：
    • 分类对比： 展示AI建议分类与原有人工分类的对比
    • 决策支持： 提供相似度评分和置信度指标辅助判断
    • 批量操作： 支持全选接受、全选拒绝、批量修改
    • 逐条确认： 可以单独审核每条任务单的分类结果
    • 分类映射： 支持将新发现的分类映射到现有Jira分类体系
    
    决策支持方式：
    • 接受AI分类： 直接使用系统建议的新分类
    • 保持原有分类： 维持Jira中原有的标签/模块分类
    • 创建混合分类： 结合AI建议和人工经验创建新的分类
    • 标记待处理： 暂时不分类，留待后续处理
    
    企业级批量处理：
    • 支持CSV/Excel格式批量导入分类决策
    • 提供分类模板下载和填写指导
    • 支持断点续传和增量导入
    • 导入进度实时显示和错误日志

5.1 场景二：新任务实时智能分流 (Real-time Classification)

    描述： 当新任务产生时，自动归集到 Phase 1 定义好的分类中。

    逻辑： 1.  通过 Webhook 接收新 Issue。
    2.  在 LanceDB 中执行 ANN（近似最近邻搜索）。
    3.  若相似度评分 > 0.8，则自动更新 Jira 的“模块”或“组件”字段。
    4.  若相似度较低，则标记为“待观察新类别”。

5.2 场景三：分类体系的动态演进

    描述： 随着业务发展，旧的分类可能失效。

    逻辑： 提供一个“重新聚类”触发机制，根据最新半年的数据调整分类中心。

7. 系统架构设计

7.1 整体架构概述

    系统采用微服务架构，主要包含以下组件：
    
    • 数据预处理服务：负责数据清洗和特征提取
    • 向量存储服务：基于LanceDB的向量数据库管理
    • 聚类分析服务：执行无监督聚类算法
    • Web管理界面：提供可视化操作界面
    • CLI工具：命令行操作接口
    • API网关：统一接口管理和认证

7.2 技术栈选型

    后端框架： FastAPI + Python 3.9+
    向量数据库： LanceDB
    嵌入模型： BGE-M3（中文优化）
    聚类算法： HDBSCAN（主）+ K-Means（备选）
    前端框架： React + Ant Design
    状态管理： Redux Toolkit
    表格组件： AG-Grid (支持大数据量展示和批量操作)
    容器化： Docker + Docker Compose
    部署： Kubernetes（推荐）
    认证： JWT Token + 简单用户名密码

7.3 部署架构

    推荐采用容器化部署方案：
    
    • 开发环境： Docker Compose 本地部署
    • 生产环境： Kubernetes 集群部署
    • 存储方案： 本地磁盘存储LanceDB数据
    • 网络配置： 内部服务通过Service通信，外部访问通过Ingress

8. 数据规范

8.1 输入数据格式

    支持两种格式的Jira导出数据：
    
    CSV格式要求：
    ```csv
    issue_id,summary,description,created_at,status,priority
    PROJ-001,"数据库连接失败","详细的问题描述...","2024-01-01 10:00:00","Open","High"
    ```
    
    JSON格式要求：
    ```json
    {
      "issues": [
        {
          "issue_id": "PROJ-001",
          "summary": "数据库连接失败",
          "description": "详细的问题描述...",
          "created_at": "2024-01-01T10:00:00Z",
          "status": "Open",
          "priority": "High"
        }
      ]
    }
    ```

8.2 数据清洗规则

    数据预处理阶段需要执行以下清洗操作：
    
    • 移除代码块：使用正则表达式匹配并移除```code```块
    • 过滤日志信息：移除包含"ERROR", "WARN", "DEBUG"等关键词的日志行
    • 清理堆栈跟踪：移除Java/Python异常堆栈信息
    • 去除HTML标签：清理富文本编辑器产生的HTML标记
    • 限制长度：Description清洗后保留前500个字符
    • 统一编码：转换为UTF-8编码

8.3 输出数据格式

    聚类结果输出格式：

9. 接口规范

9.1 RESTful API接口

    基础URL： `/api/v1`
    认证方式： Bearer Token
    
    主要接口：
    
    1. 数据上传接口
       POST `/api/v1/data/upload`
       Content-Type: multipart/form-data
       参数：file (CSV/JSON文件)
    
    2. 聚类分析接口
       POST `/api/v1/clustering/process`
       参数：{
         "algorithm": "hdbscan",
         "min_cluster_size": 10,
         "min_samples": 5
       }
    
    3. 结果查询接口
       GET `/api/v1/results/{job_id}`
    
    4. 集群详情接口
       GET `/api/v1/clusters/{cluster_id}`
    
    5. 人工复核接口
       GET `/api/v1/review/pending` - 获取待复核任务列表
       POST `/api/v1/review/batch` - 批量确认分类决策
       PUT `/api/v1/review/{issue_id}` - 单条任务复核确认
       POST `/api/v1/review/import` - 批量导入复核结果

9.2 CLI工具命令

    ```bash
    # 数据预处理
    jira-vector preprocess --input data.csv --output cleaned_data.json
    
    # 向量化处理
    jira-vector vectorize --input cleaned_data.json --model bge-m3
    
    # 聚类分析
    jira-vector cluster --algorithm hdbscan --min-size 10
    
    # 结果导出
    jira-vector export --format csv --output results.csv
    
    # 系统状态检查
    jira-vector health
    
    # 人工复核相关命令
    jira-vector review list --status pending          # 查看待复核任务
    jira-vector review confirm --all --accept         # 批量接受所有AI分类
    jira-vector review confirm --file decisions.csv   # 从文件导入复核决策
    jira-vector review export-template                # 导出复核模板
    ```
